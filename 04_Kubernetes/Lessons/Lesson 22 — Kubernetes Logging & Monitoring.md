# Lesson 22 — Kubernetes Logging & Monitoring (Observability Basics)

When your Kubernetes cluster breaks, the first question is:

> **What happened?**

Kubernetes provides three important signals:

```text
Logs
Metrics
Events
```

Together, these form the foundation of:

# Observability

---

# 1. What is Observability?

Observability means:

> Understanding system behavior from outside by analyzing available signals.

In Kubernetes:

| Signal | Meaning |
|---|---|
| Logs | What happened inside the application |
| Metrics | Resource usage and performance data |
| Events | What Kubernetes did |

---

# 2. Pod Logs

Logs are the output generated by applications running inside containers.

Examples:

```text
User login successful

Database connected

Error connecting Redis
```

---

## View Pod Logs

```bash
kubectl logs <pod-name>
```

Example:

```bash
kubectl logs nginx-pod
```

---

## Follow Logs in Real Time

Similar to:

```bash
tail -f
```

Command:

```bash
kubectl logs -f <pod-name>
```

---

## Logs From Multiple Containers

If a Pod has multiple containers:

```bash
kubectl logs <pod-name> -c <container-name>
```

---

# 3. Why Logs Matter

A Pod can be:

```text
Running ✅
```

but the application can still be:

```text
Broken ❌
```

Example:

Application error:

```text
Database connection failed
```

Logs reveal the actual application problem.

---

# 4. Kubernetes Events

Events show what Kubernetes did internally.

Examples:

```text
Scheduled Pod

Pulled Image

Started Container

Killed Container
```

---

## View Events

```bash
kubectl get events
```

---

## Describe Pod (Best Debug Method)

```bash
kubectl describe pod <pod-name>
```

It shows:

- Scheduling issues
- Image pull errors
- Container failures
- Restart reasons
- Health probe failures

---

# 5. Metrics (CPU & Memory)

Metrics answer:

> How much resource is being used?

Examples:

```text
CPU Usage

Memory Usage

Network Usage
```

---

# Metrics Server

Kubernetes uses:

```text
Metrics Server
```

to collect resource usage.

Without Metrics Server:

```bash
kubectl top
```

will not work.

---

# View Pod Usage

```bash
kubectl top pods
```

Example:

```text
NAME        CPU    MEMORY

nginx-1     10m    40Mi

nginx-2     15m    55Mi
```

---

# View Node Usage

```bash
kubectl top nodes
```

Example:

```text
Node       CPU    MEMORY

worker-1   500m   2Gi
```

---


# 10. Common Logging Problems

## Pod Crashes Immediately

Problem:

```bash
kubectl logs
```

shows nothing.

Solution:

Check previous container logs:

```bash
kubectl logs --previous <pod-name>
```

---

## Container Restart Loop

Status:

```text
CrashLoopBackOff
```

Check:

```bash
kubectl logs <pod-name>
```

and:

```bash
kubectl describe pod <pod-name>
```

---

## No Logs Showing

Possible reasons:

- Application does not write to stdout/stderr
- Wrong container selected
- Pod is not running

---

# 15. Common Mistakes

## No Metrics Server

Problem:

```bash
kubectl top
```

fails.

Solution:

Install Metrics Server.

---

## Only Checking Logs

You may miss:

```text
CPU spikes

Memory leaks

Node pressure
```

---

## Ignoring Events

Events often reveal the root cause first.

Example:

```text
FailedScheduling

ImagePullBackOff

FailedMount
```

---

# 16. Interview Questions

## What are Kubernetes logs?

Application output generated by containers.

---

## What are Kubernetes events?

System-level actions performed by Kubernetes components.

Examples:

- Scheduling
- Image pulling
- Container starting

---

## What are metrics?

Resource usage information like:

- CPU
- Memory

---

## What is Metrics Server?

A Kubernetes component that collects resource usage data.

---

## Difference Between Logs and Metrics?

| Logs | Metrics |
|---|---|
| Text output | Numeric data |
| Application behavior | Resource usage |
| Debug application issues | Monitor performance |

---

# Summary

- Observability is built using Logs, Metrics, and Events.
- Logs show application behavior.
- Metrics show resource consumption.
- Events show Kubernetes actions.
- Use `kubectl logs` for application debugging.
- Use `kubectl describe` for Kubernetes-level problems.
- Use `kubectl top` for CPU and memory usage.
- Production clusters use tools like Prometheus, Grafana, Fluent Bit, Elasticsearch, and Kibana.
